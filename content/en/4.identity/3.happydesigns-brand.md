`happydesigns/brand` is the happydesigns-specific brand layer, not the reusable identity system. Treat it as the happydesigns brand implementation; reusable identity patterns belong in `id`.

This page documents the brand layer at the ecosystem level, not as a full component or asset reference. Use the brand layer when a brand-neutral product, Docus site, or app shell needs to feel like happydesigns without changing the underlying product.

## What it provides

- Tailwind v4 theme tokens.
- Nuxt UI color-role mappings.
- CSS variables for backgrounds, text, borders, and accents.
- Typography, radius, spacing, and component defaults.
- Logo and brand primitives.
- App metadata.
- Docus and product shell styling.
- Visual QA playground.

## How it is applied

Brand-neutral products expose roles, defaults, and composition points. The brand layer fills those points with happydesigns presentation.

| Target | Stays neutral | `happydesigns/brand` applies |
| --- | --- |
| `docs` | Content model, navigation, MCP, skills, and docs behavior. | Theme tokens, Docus styling, logo usage, and visual defaults. |
| `ui` | Reusable components, content conventions, and app foundations. | Color-role values, component defaults, and brand primitives consumed by apps. |
| Business OS | Product services, permissions, workflows, and data rules. | Product shell theme, metadata, and happydesigns presentation. |
| happydesigns-owned sites | Page content and route composition. | Shared brand expression, typography, logo treatment, and layout atmosphere. |

The product remains reusable because it consumes named roles and defaults instead of hard-coding happydesigns visual decisions.

## Token use

The exact palette is a brand implementation detail. Product code should consume Nuxt UI color roles, `--ui-*` variables, and component defaults.

Use raw brand colors only inside the brand layer, logos, controlled illustrations, or deliberate brand canvas areas.

## Usage rules

- Apply happydesigns through tokens, app config, component defaults, metadata, logos, and layout composition.
- Keep reusable components in `ui` brand-neutral.
- Keep workflows and data rules in the product being themed.
- Use the brand layer to change presentation, not permissions, workflows, data rules, integrations, or server behavior.
