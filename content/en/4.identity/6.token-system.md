happydesigns styling should flow from central identity tokens into semantic component usage.

## Purpose

The token system keeps light/dark behavior, brand colors, and component styling centralized.

## Decision rule

Components consume semantic utilities. Identity layers define the palette and semantic mapping.

## Token flow

| Stage | Owns | Example |
| --- | --- | --- |
| Raw palette | Brand colors defined by the identity or brand layer. | `coral`, `petrol`, `seafoam`, `plum`, `butter`, `rose`, `sand` |
| Semantic colors | Nuxt UI color roles mapped from the palette. | `primary`, `secondary`, `success`, `info`, `warning`, `error`, `neutral` |
| CSS variables | Shared interface variables for surfaces, text, borders, radius, and accents. | `--ui-primary`, `--ui-bg`, `--ui-text`, `--ui-border` |
| Component defaults | Central Nuxt UI slot, variant, and default behavior. | buttons, cards, forms, tabs, alerts |
| Component usage | Local component markup that consumes semantic utilities. | `text-default`, `bg-elevated`, `border-muted` |

## Component rule

Components should usually consume semantic utilities such as `text-default`, `bg-elevated`, and `border-muted`. They should not choose raw brand colors unless they are part of the identity layer, a logo, or a controlled brand canvas.
