Runtime placement is the difference between reusable architecture and duplicated product behavior. The same business action should not be reimplemented separately for UI, API, jobs, webhooks, and MCP.

Use this page when a change could be written in several places. The goal is to place behavior once and expose it through the right channels.

Put deterministic behavior below the channel. Put request-specific behavior at the runtime edge. Put presentation in the Nuxt app or layer.

## Runtime model

```mermaid
flowchart LR
  UI["Nuxt UI"] --> Service["Application service"]
  API["HTTP API"] --> Service
  Job["Scheduled job"] --> Service
  Webhook["Webhook"] --> Service
  MCP["MCP tool"] --> Service
  Service --> Domain["Domain package"]
  Service --> Audit["Audit and policy"]
```

## Runtime ownership

Use this ownership table when choosing where a behavior belongs:

| Runtime | Owns |
| --- | --- |
| Plain TypeScript | Domain rules, validation, calculations, parsing, formatting, deterministic workflows. |
| Application service | Commands, queries, policy checks, orchestration, channel-independent behavior. |
| Nitro | HTTP handlers, server plugins, deployment bindings, runtime adapters, scheduled jobs. |
| Nuxt module | Installation, hooks, generated files, runtime config, typed integration. |
| Nuxt layer | UI composition, layout, app config, content, brand and customer presentation. |
| Nuxt app | Concrete product experience and route-level composition. |

## Anti-patterns

UI visibility is not authorization. Extension metadata is not permission enforcement. Demo API boundaries are not production authorization.
