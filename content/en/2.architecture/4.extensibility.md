happydesigns reuses Nuxt standards before inventing custom extension systems.

Extensibility means a product has clear places where apps, layers, modules, or integrations can customize it. It does not mean every customer-specific change becomes reusable infrastructure.

## Extension order

Use the platform feature that matches the change:

1. Presentation, branding, layout, pages, components, or app config: use a Nuxt layer.
2. Reusable Nuxt setup, hooks, runtime config, generated files, generated types, or server handlers: use a Nuxt module.
3. Deterministic domain logic: use a plain TypeScript package.
4. Request, runtime, or deployment integration: use Nitro.
5. External runtime access: use API, webhook, OAuth, or MCP.
6. Product-specific customization: add a narrow customization point in the product that owns the behavior.

## Product customization points

A product customization point is part of the product, not a hidden customer override. It should define:

- What can be customized.
- Which app, layer, module, API, or content file provides the customization.
- What data the customization receives.
- Which product still owns validation, permissions, storage, and workflow state.

Customization points do not turn presentation metadata into permission to perform product actions:

- Private internals as public extension APIs.
- UI flags as authorization.
- Extension metadata as permission enforcement.
- Customer layers that change product behavior.
