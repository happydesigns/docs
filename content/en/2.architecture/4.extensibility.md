# Extensibility

happydesigns should reuse Nuxt standards before inventing custom plugin systems.

## Decision flow

1. Is it presentation, branding, layout, pages, components, or app config? Use a Nuxt layer.
2. Is it installable behavior, server routes, hooks, runtime config, generated files, or typed integration? Use a Nuxt module.
3. Is it deterministic domain logic? Use a plain TypeScript workspace package.
4. Is it request, runtime, or deployment integration? Use Nitro.
5. Is it external runtime access? Use API, webhook, OAuth, or MCP.
6. If none of those are enough, add a small documented product extension contract.

## Extension rules

- Customer extensions must not depend on private internals.
- UI visibility is not authorization.
- Extension metadata is not permission enforcement.
- Sensitive actions need reviewed service contracts.
- Business OS should not invent a custom plugin system where Nuxt layers, Nuxt modules, Nuxt hooks, Nitro routes, runtime config, or app config already solve the problem.
