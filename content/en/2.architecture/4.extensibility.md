happydesigns reuses Nuxt standards before inventing custom extension systems.

Extensibility makes the system safer to adapt. It does not create private plugin APIs, bypass authorization, or move product behavior into customer presentation layers.

## Extension order

Use platform extension points first.

1. Presentation, branding, layout, pages, components, or app config: use a Nuxt layer.
2. Installable behavior, server routes, hooks, runtime config, generated files, or typed integration: use a Nuxt module.
3. Deterministic domain logic: use a plain TypeScript package.
4. Request, runtime, or deployment integration: use Nitro.
5. External runtime access: use API, webhook, OAuth, or MCP.
6. Anything else: add a small documented product extension contract.

## Extension contract scope

Good extension contracts define what can be customized without exposing private internals:

- Stable extension contracts.
- Customer-safe customization boundaries.
- Nuxt-native integration choices.
- Explicit rules for sensitive actions.

They must not turn presentation metadata into permission to perform product actions:

- Private internals as public extension APIs.
- UI flags as authorization.
- Extension metadata as permission enforcement.
- Customer layers that change business behavior.
