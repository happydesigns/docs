Separation of concern is the main quality rule for happydesigns architecture.

Use this page when a change feels like it could fit in more than one product or layer. The table gives the first placement decision; the surrounding docs refine it.

Do not let a convenient implementation location become an accidental owner.

| Question | Put it where |
| --- | --- |
| Is it reusable visual or content infrastructure? | `happydesigns/ui` |
| Is it a brand token, logo, motif, or brand layout default? | `identity` or a brand layer |
| Is it customer-specific copy or page content? | Customer project |
| Is it product behavior? | Product package or module |
| Is it deterministic business logic? | Plain TypeScript package |
| Is it API, MCP, or webhook behavior? | Server and application-service contract |
| Is it temporary planning? | `.agents/` or Business OS, not published docs |
| Is it customer instruction? | Future help docs |

## Brand-layer boundary

Presentation, assets, metadata, layout motifs, app config, and customer-facing copy.

Domain behavior, authorization, tenant isolation, audit behavior, financial behavior, credentials, APIs, webhooks, MCP tools, production logging runtime, or public write behavior.
