happydesigns uses layered architecture so product logic, runtime access, UI foundations, brand expression, and project composition do not collapse into one place.

This section is the decision layer for where code belongs. It should help a contributor decide whether a change belongs in a plain TypeScript package, Nitro runtime, Nuxt module, Nuxt layer, product app, identity layer, docs site, or customer help center.

Put stable reusable behavior as low as practical. Put project-specific composition as high as practical.

Read the architecture stack from reusable foundation to concrete experience:

1. Plain TypeScript packages own domain logic and contracts.
2. Nitro and server services own runtime boundaries, jobs, APIs, and adapters.
3. Nuxt modules own installable behavior.
4. Nuxt layers own UI, brand expression, layouts, pages, and app config.
5. Apps and projects own the actual experience.

## Scope

This section owns the shared architecture decisions:

- Product and layer boundaries.
- Runtime placement rules.
- Reuse and extension strategy.
- Brand-neutral product-core policy.
- Practical placement examples.

It does not replace product or customer documentation:

- Full product implementation references.
- Customer instructions.
- Temporary planning logs.
- Product-specific API contracts unless they affect ecosystem boundaries.
