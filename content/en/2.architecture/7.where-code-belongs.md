Use this page when a change could fit in more than one place.

## Purpose

The examples turn architecture rules into day-to-day placement decisions. They are intentionally concrete so agents can choose the right owner before editing.

## Decision rule

Put the most stable reusable part at the lowest useful level. Put project-specific composition at the project level. Do not move code into a reusable package only because it might be useful later.

| Change | Belongs in |
| --- | --- |
| New reusable content card pattern | `happydesigns/ui` |
| happydesigns logo component | `happydesigns/brand` or an identity layer |
| Client logo and client colors | Client identity layer |
| Customer homepage text | Customer site content |
| Invoice finalization logic | Business OS service or domain package |
| App shell navigation policy | Product app or reviewed module |
| Nuxt UI color mapping | Identity or brand layer |
| Customer-specific dashboard widget | Nuxt layer or reviewed extension point |
| n8n integration behavior | Business OS API, webhook, or integration module |
| Documentation authoring rule | `happydesigns/docs` |
| Customer editing guide | Future `happydesigns/help` |

## Does not belong

- Product behavior in brand layers.
- Brand-specific identity in `ui`.
- Temporary planning notes in published docs.
- Customer support guides in architecture docs.
