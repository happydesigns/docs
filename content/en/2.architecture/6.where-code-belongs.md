Use this page when a change could fit in more than one place.

The examples turn architecture rules into day-to-day placement decisions.

Put shared code where every caller can use it. Put project-specific composition in the project. Do not move code into a reusable package only because it could be reused someday.

## Placement examples

| Change | Belongs in |
| --- | --- |
| New reusable content card pattern | `happydesigns/ui` |
| happydesigns logo component | `happydesigns/brand` or an identity layer |
| Client logo and client colors | Client identity layer |
| Customer homepage text | Customer site content |
| Invoice finalization logic | Business OS service or plain TypeScript package for business rules |
| App shell navigation policy used by one product | Product app |
| Shared app shell navigation package | Nuxt module or product package |
| Nuxt UI color mapping | Identity or brand layer |
| Customer-specific dashboard layout | Customer project or client Nuxt layer |
| Reusable dashboard behavior | Product package that owns the dashboard |
| n8n integration behavior | Business OS API, webhook, or integration module |
| Documentation authoring rule | `happydesigns/docs` |
| Customer editing guide | `happydesigns/help` |

## Does not belong

- Product behavior in brand layers.
- Brand-specific identity in `ui`.
- Temporary planning notes in published docs.
- Customer support guides in architecture docs.
