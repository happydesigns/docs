Separation of concern is the main quality rule for happydesigns architecture. Use this page when a change feels like it could fit in more than one product or layer.

The table gives the first placement decision; the surrounding docs refine it. Do not let a convenient implementation location become an accidental owner.

## Placement decisions

| Question | Put it where |
| --- | --- |
| Is it reusable visual or content infrastructure? | `happydesigns/ui` |
| Is it a brand token, logo, motif, or brand layout default? | `id` or a brand layer |
| Is it customer-specific copy or page content? | Customer project |
| Is it product logic? | Product package or module |
| Is it reusable business logic with the same result for the same input? | Plain TypeScript package |
| Is it API, MCP, or webhook behavior? | Server route and application service |
| Is it temporary planning? | `.agents/` or Business OS, not published docs |
| Is it customer instruction? | Customer help docs |

## Brand-layer boundary

A brand layer may own presentation, assets, metadata, layout motifs, app config, and customer-facing copy.

A brand layer themes the product. It does not change permissions, workflows, data rules, integrations, storage, or server behavior.
