Architecture stays useful only when boundaries are enforced.

| Question | Put it where |
| --- | --- |
| Is it reusable visual or content infrastructure? | `ui` |
| Is it a brand token, logo, motif, or brand layout default? | `id` / brand layer |
| Is it customer-specific copy or page content? | Customer project |
| Is it product behavior? | Product package or module |
| Is it deterministic business logic? | Plain TypeScript package |
| Is it API, MCP, or webhook behavior? | Server and application-service contract |
| Is it temporary planning? | `.agents/` or Business OS, not published docs |
| Is it customer instruction? | Future help docs |

## Brand layers may own

- Presentation.
- Assets.
- Metadata.
- Layout motifs.
- App config.
- Customer-facing copy.

## Brand layers must not own

- Domain behavior.
- Authorization.
- Tenant isolation.
- Audit behavior.
- Financial behavior.
- Credentials.
- APIs, webhooks, or MCP tools.
- Production logging runtime.
- Public write behavior.
