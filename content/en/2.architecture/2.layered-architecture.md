happydesigns separates reusable foundations, installable behavior, identity, and concrete project work.

## Purpose

Layering lets products share structure without sharing the wrong responsibilities. It also gives agents a reliable map before changing code.

## General model

| Layer | Owns | Avoid putting here |
| --- | --- | --- |
| Plain TypeScript | Types, rules, calculations, parsing, validation, deterministic services, and contracts. | Request context, presentation, deployment bindings. |
| Nitro | HTTP handlers, server plugins, jobs, adapters, deployment bindings, and runtime context. | Reusable domain rules or UI composition. |
| Nuxt modules | Installable behavior, hooks, generated files, typed integration, and runtime config. | Brand expression or project-specific pages. |
| Nuxt layers | UI, branding, pages, layouts, components, app config, and content conventions. | Channel-independent business behavior. |
| Host app or project | The concrete product experience and final route-level composition. | Shared product logic that belongs lower. |

## Website and docs model

1. `happydesigns/ui` provides reusable interface and content foundations.
2. Identity and brand layers provide tokens, logos, app config, and visual expression.
3. The site or product project composes content, pages, routes, and final experience.

## Business OS model

Business behavior should live below every runtime channel:

1. Domain packages own entities, policies, deterministic rules, and documents.
2. Application services own commands, queries, workflows, and orchestration.
3. Runtime channels expose the same services through UI, API, MCP, jobs, and webhooks.
4. The Nuxt UI app provides the human operations surface.
5. Brand and customer layers change presentation only.

## Placement rule

Use the lowest-level runtime that preserves product quality. For channel-specific placement examples, read [Runtime placement](/en/architecture/runtime-placement).
