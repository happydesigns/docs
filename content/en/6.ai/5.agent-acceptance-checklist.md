The acceptance checklist makes agent work reviewable. It is the minimum standard before an agent reports a change as complete.

Use this checklist for code and docs changes. It catches wrong ownership, stale names, broken navigation, and unvalidated assumptions.

Do not finish with only a prose summary. Verify the changed surface.

## Checklist

- The change has a clear product, layer, module, app, or docs owner.
- The implementation does not move business logic into brand or identity layers.
- Reusable UI remains brand-neutral unless the changed package is a brand layer.
- Package names use repository metadata.
- Documentation uses lowercase `happydesigns`.
- Page titles come from Docus frontmatter and are not duplicated as first Markdown headings.
- Navigation has no duplicate section-title pages.
- Mermaid diagrams render or the build fails visibly.
- Skills point to published docs paths.
- Build or prepare commands have run, or the limitation is documented.

## Search checks

Run targeted searches for stale names, duplicate headings, and old content folders before finishing.
