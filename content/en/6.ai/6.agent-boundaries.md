Agents preserve boundaries even when a shortcut would make the immediate task easier. This page lists actions that create long-term architecture debt or publish the wrong kind of information.

When unsure, stop and identify the product boundary before editing.

## Do not

- Use capitalized brand variants; write `happydesigns`.
- Put business logic in brand layers.
- Put brand-specific code in `ui`.
- Duplicate Nuxt, Nuxt UI, Docus, or Nuxt Studio docs.
- Scatter raw Tailwind colors and `dark:` variants through components.
- Use UI visibility as authorization.
- Expose private Business OS details in public docs unless they are stable operating docs.
- Publish `.agents/` task notes as documentation.
- Treat package metadata as permission to load runtime extensions.
- Treat development or demo API boundaries as production authorization.
- Turn customer help into technical developer docs.
- Over-document unstable implementation details.

## If unsure

Read the architecture docs and source-of-truth page. If the boundary is still unclear, ask for human review.
