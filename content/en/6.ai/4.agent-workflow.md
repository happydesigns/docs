Agents should follow this workflow before changing happydesigns code or docs.

1. Inspect the repo and identify which happydesigns product, layer, or project is being changed.
2. Read relevant docs through Docus or MCP.
3. Determine where the change belongs.
4. Reuse existing layers, components, and config before creating new code.
5. Preserve separation of concern.
6. Implement minimal, focused changes.
7. Validate with available commands.
8. Do not introduce new architecture unless the docs justify it.
9. If unsure, add a small docs TODO or ask for human review instead of guessing product direction.

## Default bias

Keep durable architecture in docs. Keep temporary task planning in `.agents/` or Business OS. Do not publish scratch state as documentation.
