Use this page after the ecosystem overview when a request could fit in more than one place. The goal is to keep customer-specific work close to the customer project and shared work in the foundation that owns it.

## Choose the destination

A request can touch several areas, but one place defines the shared part. Start by deciding what kind of work this is:

| Work | Belongs in | Why |
| --- | --- | --- |
| One customer site, app, UI, or content change. | That application or customer layer. | The work is local to one customer or one application. |
| Shared UI, content conventions, or app foundations. | `ui`. | The pattern supports multiple Nuxt apps or websites. |
| Identity model, brand-layer structure, or token flow. | `id`. | The pattern supports happydesigns and client identities. |
| happydesigns visual defaults. | `brand`. | The work is specific to the happydesigns brand. |
| Product behavior such as permissions, workflows, or domain rules. | The product package or service that owns it. | The behavior must stay consistent across every interface that exposes the product. |
| Shared layout variant behavior. | `nuxt-variants`. | The same layout option is reused by multiple pages or apps. |
| Customer instructions. | `help`. | The reader is a customer operating a website or app. |
| Ecosystem architecture. | `docs`. | The decision affects how products, foundations, or docs fit together. |

## Choose the implementation level

After the destination is clear, implementation work still needs the right technical level. Choose the level that keeps shared behavior reusable and app-specific composition in the app:

1. Plain TypeScript for shared rules, calculations, and contracts.
2. Nitro for server-side behavior such as routes, APIs, jobs, and deployment bindings.
3. Nuxt modules for installable behavior and generated integration.
4. Nuxt layers for UI, branding, pages, layouts, and app config.
5. Nuxt apps for concrete application and customer-project composition.

## Decision examples

These examples compare one-project work with reusable foundation work:

| Request | Belongs in | Why |
| --- | --- | --- |
| Add a reusable content card pattern used by several sites. | `ui` | The pattern is shared interface infrastructure, not one project composition. |
| Add a client logo and color palette. | Client identity layer | The change is presentation and identity, not product logic. |
| Add invoice finalization rules. | Business OS domain or service layer | The rule must stay consistent across every surface that uses it. |
| Add a layout option used by one app. | App layer | Only one app needs it. |
| Add a named layout variant used by several apps. | `nuxt-variants` | The variant is reusable app infrastructure. |
| Explain how a customer edits website content. | `help` | Customer instruction is not ecosystem architecture. |

## When unsure

Keep the work in the most specific place that solves the current problem. Move it into a reusable foundation only when reuse is real, or when the foundation itself is being defined.

Update this manual when the decision changes how products, foundations, or docs fit together. For implementation detail, update the product docs or repository docs that own the detail. If docs, code, or task notes disagree, use [Source of truth](/en/start/source-of-truth).
