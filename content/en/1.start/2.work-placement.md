Use this page after the ecosystem overview when new work or an existing change could fit in more than one place. The goal is to keep customer-specific work close to the customer project and shared work in the foundation that can reuse it.

## Choose the destination

Start by deciding what kind of work this is:

| Work | Belongs in | Why |
| --- | --- | --- |
| Customer-specific app, site, UI, or content work. | That application or customer layer. | The change is local to one customer or one application. |
| Shared UI, content conventions, or app foundations. | `ui`. | The pattern supports multiple Nuxt apps or websites. |
| Identity and brand-layer structure. | `id`. | The pattern supports happydesigns and client identities. |
| happydesigns visual defaults. | `brand`. | The change is specific to the happydesigns brand. |
| Product rules or workflows. | The product package or service that owns them. | The rule must be shared by every interface that exposes the product. |
| Shared layout variant behavior. | `nuxt-variants`. | The same layout option is reused by multiple pages or apps. |
| Customer instructions. | `help`. | The reader is a customer operating a website or app. |
| Ecosystem architecture. | `docs`. | The decision affects how products, foundations, or docs fit together. |

## Choose the implementation level

Use this section when creating or changing implementation. Choose the lowest level that keeps shared work reusable and app-specific work in the app:

1. Plain TypeScript for shared rules, calculations, and contracts.
2. Nitro for server routes, APIs, jobs, and deployment bindings.
3. Nuxt modules for installable behavior and generated integration.
4. Nuxt layers for UI, branding, pages, layouts, and app config.
5. Nuxt apps for concrete application and customer-project composition.

## Decision examples

Use examples to compare common cases. A request can touch several places, but one place should carry the reusable part.

| Request | Belongs in | Why |
| --- | --- | --- |
| Add a reusable content card pattern used by several sites. | `ui` | The pattern is shared interface infrastructure, not one project composition. |
| Add a client logo and color palette. | Client identity layer | The change is presentation and identity, not product logic. |
| Add invoice finalization rules. | Business OS domain or service layer | The rule must be shared by UI, API, jobs, webhooks, and MCP. |
| Add a layout option used by one app. | App layer | The choice is local composition. |
| Add a named layout variant used by several apps. | `nuxt-variants` | The variant is reusable app infrastructure. |
| Explain how a customer edits website content. | `help` | Customer instruction is not ecosystem architecture. |

## When unsure

Keep the work in the most specific place that solves the problem. Move it into a reusable foundation only when more than one application needs the same pattern, or when the foundation itself is being defined.

If the decision affects how products, foundations, or docs fit together, update this manual. If it only explains implementation detail, update the product docs or repository docs.
