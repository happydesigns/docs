World-class docs are useful only when readers know which source wins. happydesigns keeps stable strategy in published docs and unstable work state in operational tools.

Published docs describe stable product strategy and architecture. Source code describes implementation state. `.agents/` and task logs describe work in progress.

## Which source wins

| Question | Source of truth |
| --- | --- |
| What is the implementation state? | The product repository code and package metadata. |
| What is the shared architecture rule? | `docs.happydesigns.de`. |
| How does an agent act? | Published skills plus relevant docs pages. |
| What is the product-specific implementation guide? | The dedicated product docs or repository docs. |
| What is temporary task state? | `.agents/`, Business OS, or issue/PR context, not published docs. |
| What do customers read? | `help.happydesigns.de`. |

## When sources conflict

Use repository metadata for package names and code facts. Use these docs for ecosystem boundaries. Update the docs when a product decision becomes stable.

## Examples

- Repository metadata and package registries define package names; docs use those names.
- `happydesigns/brand` is the happydesigns-specific brand layer; it is not the reusable Identity product.
- Business OS details belong here only when they explain ecosystem strategy.
