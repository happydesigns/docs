After the ecosystem map and placement rules, the remaining question is which source to trust. Different sources answer different questions: published docs explain strategy and architecture, repositories show code and package metadata, and `.agents/` or task logs show work in progress.

## Which source wins

| Question | Trust first |
| --- | --- |
| What is implemented? | Product repository code and package metadata. |
| What is the shared architecture rule? | `docs.happydesigns.de`. |
| What do agents follow? | Published skills plus the relevant docs pages. |
| How does one product work internally? | Dedicated product docs or repository docs. |
| What is being worked on right now? | `.agents/`, Business OS, or issue/PR context. |
| What do customers read? | `help.happydesigns.de`. |

## When sources conflict

Treat conflicts as a signal that either docs or code need updating. Use repository metadata for concrete implementation facts, and use these docs for ecosystem boundaries and intent.

When implementation changes how products, foundations, or docs fit together, update this manual. When it only changes how one product works internally, update that product's docs or repository docs.

## Examples

- Repository metadata and package registries define package names; docs use those names.
- `id` owns the identity model and brand-layer mechanism; `brand` applies the happydesigns defaults.
- Business OS details belong here when they change the wider ecosystem model. Product-specific behavior belongs in the Business OS docs or repository docs.
