Use this page after the operating model and product map when work could belong in several places. First choose the destination: the app, foundation, product, or docs site that owns the result. Then choose the technical level inside that destination.

Keep client-specific work close to the client project. Move work into a foundation only when it creates a reusable rule, pattern, or capability.

## Choose the destination

A request can touch several areas, but one place owns the main decision. Ask what the work changes first. If the product owner is unclear, use the [Product map](/start/product-map).

| Work changes | Belongs in | Why |
| --- | --- | --- |
| One client website, app screen, page, or content set. | That application, site, or client brand layer. | The work is local to one client or one application. |
| Shared UI components, content structures, or app foundations. | [`ui`](/products/ui). | The pattern supports multiple Nuxt apps or websites. |
| Identity model, brand-layer mechanism, or token flow. | [`id`](/products/id). | The work defines reusable identity and theming behavior. |
| happydesigns brand defaults. | [`brand`](/products/brand). | The work applies the happydesigns identity without changing product behavior. |
| Product behavior such as permissions, workflows, or domain rules. | The product package or service that owns it. | The behavior must stay consistent across every interface that exposes the product. |
| Shared layout variant behavior. | [`nuxt-variants`](/products/nuxt-variants). | The same layout option is reused by multiple pages or apps. |
| Site or app operating instructions. | [`help`](/products/help). | The reader is operating a delivered website or app. |
| Ecosystem architecture or documentation policy. | [`docs`](/products/docs). | The decision affects how products, foundations, or docs fit together. |

## Choose the implementation level

After the destination is clear, choose the level inside it. A one-page content change can stay in content. A reusable presentation pattern may belong in a layer. Installable Nuxt setup belongs in a module. Product rules belong below the UI that exposes them.

Use [Runtime placement](/architecture/runtime-placement) for the full ownership model.

As a quick check:

1. Product rules that must behave the same across UI, API, jobs, webhooks, or MCP belong below those interfaces.
2. Installable Nuxt behavior belongs in a module.
3. Presentation, layout, content, and brand composition belong in layers or content.
4. One app's pages, routes, and project composition belong in that app.

## Decision examples

These examples compare local project work with reusable foundation work:

| Request | Belongs in | Why |
| --- | --- | --- |
| Add a content card pattern used by several sites. | [`ui`](/products/ui) | The pattern is shared interface infrastructure. |
| Add a logo and tokens for one client brand. | Client brand layer | The change is presentation and identity, not product logic. |
| Add invoice finalization rules. | [Business OS](/products/os) domain or service layer | The rule must stay consistent across every surface that uses it. |
| Add a layout option used by one app. | That app | Only one app needs it. |
| Add a named layout variant used by several apps. | [`nuxt-variants`](/products/nuxt-variants) | The variant is reusable app infrastructure. |
| Explain how site editors update website content. | [`help`](/products/help) | Operating guidance is not ecosystem architecture. |
| Explain why applications build on foundations. | [`docs`](/products/docs) | The topic changes how the ecosystem is understood. |

## When unsure

Start specific. Keep the work in the project that needs it until there is a real reusable shape. Promote it into a foundation only when another project needs the same capability, or when the foundation itself is being deliberately defined.

Update this manual when the decision changes how products, foundations, or docs fit together. For implementation detail, update the product docs or repository docs that own the detail. If docs, code, or task notes disagree, use [Source of truth](/start/source-of-truth).
