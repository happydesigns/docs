World-class docs are useful only when readers know which source wins. happydesigns should keep durable strategy in published docs and unstable work state in operational tools.

Published docs describe stable product strategy and architecture. Source code describes implementation state. `.agents/` and task logs describe work in progress.

## Authority order

| Question | Source of truth |
| --- | --- |
| What is the implementation state? | The product repository code and package metadata. |
| What is the shared architecture rule? | `docs.happydesigns.de`. |
| How should an agent act? | Published skills plus relevant docs pages. |
| What is the product-specific implementation guide? | The dedicated product docs or repository docs. |
| What is temporary task state? | `.agents/`, Business OS, or issue/PR context, not published docs. |
| What should customers read? | Future `help.happydesigns.de`. |

## When sources conflict

Use repository metadata for package names and code facts. Use these docs for ecosystem boundaries. Update the docs when a product decision becomes durable.

## Examples

- `happydesigns/ui` publishes `@happydesigns/ui`; docs should use that name.
- `happydesigns/brand` is the concrete brand layer; it is not the generic identity product.
- Business OS details belong here only when they explain ecosystem strategy.
