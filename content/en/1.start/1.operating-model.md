happydesigns uses reusable foundations so applications do not become one-off project code. Each part has a clear job and a small set of places where other projects connect to it. This page helps place changes in the right package, Nuxt module, Nuxt layer, app, or docs site.

## Two placement questions

Start with the result, not the file path.

First, choose who owns the outcome:

- An application or customer layer for one concrete app or site.
- A reusable foundation for shared UI, identity, content conventions, project setup, or product rules.
- `help` for customer instruction.
- `docs` for ecosystem architecture.

Then choose where the implementation lives inside that owner or foundation. Use the lowest level that can carry the change without making local app code responsible for shared work:

1. Plain TypeScript for shared rules, calculations, and contracts.
2. Nitro for server routes, APIs, jobs, and deployment bindings.
3. Nuxt modules for installable behavior and generated integration.
4. Nuxt layers for UI, branding, pages, layouts, and app config.
5. Nuxt apps for concrete application and customer-project composition.

## Decision examples

Use examples to check both decisions before editing. A request can touch several places, but one place should carry the reusable part.

| Request | First owner | Why |
| --- | --- | --- |
| Add a reusable content card pattern used by several sites. | `happydesigns/ui` | The pattern is shared interface infrastructure, not one project composition. |
| Add a client logo and color palette. | Client identity layer | The change is presentation and identity, not product logic. |
| Add invoice finalization rules. | Business OS domain or service layer | The behavior must be shared by UI, API, jobs, webhooks, and MCP. |
| Add a layout option used by one app. | App layer | The choice is local composition. |
| Add a named layout variant used by several apps. | `happydesigns/nuxt-variants` | The variant is reusable app infrastructure. |
| Explain how a customer edits website content. | `happydesigns/help` | Customer instruction is not ecosystem architecture. |

## Operating loop

1. Identify who owns the outcome.
2. Read the published docs through MCP or local files.
3. Choose the implementation level that should carry the change.
4. Make the smallest complete change.
5. Validate with the checks that match the risk.
6. Update published docs when a rule changed.

## Good output

A good change makes the next change easier. It clarifies ownership, reduces duplication, and keeps brand expression, product logic, interfaces, and customer instructions separate.
