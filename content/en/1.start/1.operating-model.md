happydesigns builds customer websites, client brand systems, and apps such as [Business OS](/products/os) on shared foundations. The goal is to solve a problem once, make the solution reusable, and apply it across projects.

Start here to understand what happydesigns builds. Then use [Work placement](/start/work-placement) to decide where new work belongs and [Source of truth](/start/source-of-truth) to decide which source to trust.

## What happydesigns builds

At the highest level, the ecosystem has three kinds of output:

| Area | Role | Examples |
| --- | --- | --- |
| Applications | Products and customer-facing projects assembled from foundations. | Customer sites, [`os`](/products/os). |
| Foundations | Reusable packages, layers, and modules that make applications and brand systems consistent. | [`ui`](/products/ui), [`id`](/products/id), [`brand`](/products/brand), [`nuxt-variants`](/products/nuxt-variants). |
| Documentation | Docs for different readers and levels of detail. | Product docs, [`docs`](/products/docs), [`help`](/products/help). |

Foundations hold the shared parts that would otherwise be rebuilt in each project. The documentation split keeps ecosystem strategy, product implementation, and customer guidance in the right place.

For individual product roles, repository links, and package links, use the [Product map](/products/product-map).

## Build map

Read this map from top to bottom. Foundations feed applications; product docs, [`docs`](/products/docs), and [`help`](/products/help) are not drawn as dependencies because they explain the system rather than build it.

```mermaid
    ID --> ClientIdentity["client identity layers"]
  end

  subgraph Applications["Applications"]

    OS["os<br/>Business OS"]
  end

  UI --> Sites
  UI --> OS
  Brand --> Sites
  Brand --> OS
  ClientIdentity --> Sites
```
