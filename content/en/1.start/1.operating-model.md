The happydesigns ecosystem is organized around reusable products that last across projects, not one-off project code. Each product has a clear owner, a stable boundary, and a small set of integration points.

The operating model keeps humans and agents aligned. A reader can inspect a task, identify the product boundary, choose the right Nuxt or TypeScript layer, and avoid moving behavior into the wrong place.

## Runtime placement order

Choose the lowest layer that can own the change without hurting product quality. Stop at the first level that fits:

1. Plain TypeScript for shared rules, calculations, and contracts.
2. Nitro for server routes, APIs, jobs, and deployment bindings.
3. Nuxt modules for installable behavior and generated integration.
4. Nuxt layers for UI, branding, pages, layouts, and app config.
5. Nuxt apps for concrete user experiences.

## Decision examples

Use examples to check the level before editing. The same request can involve several layers, but one layer owns the reusable behavior.

| Request | First owner | Why |
| --- | --- | --- |
| Add a reusable content card pattern used by several sites. | `happydesigns/ui` | The pattern is shared interface infrastructure, not one project composition. |
| Add a client logo and color palette. | Client identity layer | The change is presentation and identity, not product behavior. |
| Add invoice finalization rules. | Business OS domain or service layer | The behavior must be shared by UI, API, jobs, webhooks, and MCP. |
| Add a layout option used by one app. | App layer | The choice is local composition. |
| Add a named layout variant used by several apps. | `happydesigns/nuxt-variants` | The variant is reusable app infrastructure. |
| Explain how a customer edits website content. | `happydesigns/help` | Customer instruction is not ecosystem architecture. |

## Operating loop

1. Identify the product boundary.
2. Read the published docs through MCP or local files.
3. Choose the package, module, layer, app, or docs site that owns the change.
4. Make the smallest coherent change.
5. Validate with the checks that match the risk.
6. Update published docs only when a stable rule changed.

## Good output

A good change leaves the ecosystem easier to reason about. It makes ownership clearer, reduces duplicated behavior, and keeps brand expression, product logic, runtime access, and customer instructions separate.
