The happydesigns ecosystem is organized around durable reusable products, not one-off project code. Each product should have a clear owner, a stable boundary, and a small set of integration points.

## Purpose

The operating model keeps humans and agents aligned. A reader should be able to inspect a task, identify the product boundary, choose the right Nuxt or TypeScript layer, and avoid moving behavior into the wrong place.

## Decision rule

Use the lowest-level runtime that preserves product quality. Choose in this order, and stop at the first level that fits the change:

1. Plain TypeScript for deterministic logic and shared contracts.
2. Nitro for runtime boundaries, APIs, jobs, and deployment bindings.
3. Nuxt modules for installable behavior and generated integration.
4. Nuxt layers for UI, branding, pages, layouts, and app config.
5. Nuxt apps for concrete user experiences.

## Operating loop

1. Identify the product boundary.
2. Read the current docs through MCP or local files.
3. Choose the package, module, layer, app, or docs surface that owns the change.
4. Make the smallest coherent change.
5. Validate with the checks that match the risk.
6. Update durable docs only when a stable rule changed.

## Good output

A good change should leave the ecosystem easier to reason about. It should make ownership clearer, reduce duplicated behavior, and keep brand expression, product logic, runtime access, and customer instructions separate.
