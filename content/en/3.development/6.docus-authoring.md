Docus is the documentation medium for both humans and agents. Its structure should make decisions easy to find and stable enough to retrieve through MCP and LLM files.

Use this page before adding or restructuring documentation. It keeps authoring consistent across pages and prevents Docus-specific mistakes.

Use frontmatter for the page title and description. Do not repeat the same title as the first Markdown heading.

## Page pattern

- Frontmatter `title` and `description`.
- Opening prose that explains context, purpose, and reader relevance.
- A section heading before substantive lists, tables, diagrams, code blocks, or decision models.
- A clear rule sentence when the page needs a decision boundary.
- Headings only for real content groups such as `Scope`, `Examples`, `Model`, or `Policy`.
- Diagrams, tables, lists, or cards only when they make the page easier to understand.

## Page types

Choose the page type before choosing headings. The type should tell the author what kind of reader decision the page supports.

| Page type | Use it for | Typical sections |
| --- | --- | --- |
| Landing page | Routing readers to the right part of the manual. | Reader paths, section cards, scope statement. |
| Model page | Explaining how important parts relate. | Model, ownership, examples, anti-patterns. |
| Boundary page | Separating similar products, layers, or responsibilities. | Scope, examples, does not own. |
| Workflow page | Explaining a repeatable sequence. | Ordered steps, checks, good output. |
| Policy page | Recording durable rules that affect many repositories. | Rule, allowed patterns, avoided patterns, examples. |
| Product overview | Explaining the ecosystem role of a product. | Role, what it provides, boundary. |

Do not force every page into the same shape. A short boundary page can be mostly prose and a table. A workflow page should usually use an ordered list. A model page may use a diagram if the prose needs it.

## Human readability

Write pages as documentation, not forms. Short pages should not repeat the same template sections just because other pages have them. Fold purpose into the introduction, then use prose to connect rules, examples, diagrams, and boundaries.

## Link policy

Rely on the aside navigation and Docus previous/next controls for sequential reading. Do not add a repeated `## Read next` section to every page.

Use inline links in prose when the target page directly supports the sentence. For example, link to [Runtime placement](/en/architecture/runtime-placement) when explaining channel-specific ownership, not as a generic footer item.

## Medium choice

| Use | When |
| --- | --- |
| Prose | Explaining intent, tradeoffs, and policy. |
| Ordered lists | Showing a sequence, priority order, or operating loop. |
| Tables | Comparing ownership, decisions, or examples. |
| Mermaid | Showing topology, convergence, branching, feedback loops, or ownership flows. |
| MDC cards | Landing pages and small navigation surfaces. |
| Code blocks | File structure, commands, or exact configuration examples. |

## Diagram policy

Use Mermaid when a diagram explains structure that prose, a list, or a table would hide:

- Product topology with multiple relationships.
- Runtime channels converging into shared services.
- Branching identity or agent workflows.
- Feedback loops or ownership flows.

Every diagram section needs prose that explains how to read the diagram and what decision it supports. Do not make the diagram carry the entire section by itself.

Do not use Mermaid for a simple sequence such as `A -> B -> C`. Use an ordered list when order matters, or a table when comparing ownership.

## i18n

English is the canonical language for the published operating manual. German remains configured but should not receive copied English pages. Add German content only when it is intentionally translated.

## Navigation model

Prefer Docus default aside navigation when the documentation is one operating manual with one primary reader set. This keeps the full structure visible and makes nearby pages easier to discover.

Use Docus sub-navigation only when the documentation benefits from a real third layer:

- Header section: a major documentation mode, product area, or persona.
- Aside section: the local topic tree within that mode.
- Page: the actual decision, guide, or reference page.

Do not use header sub-navigation only to make the top of the page look more structured. It should solve a real information architecture problem, such as separating distinct personas or preventing one aside from becoming overloaded.
