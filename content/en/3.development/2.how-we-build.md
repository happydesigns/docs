happydesigns projects should be easy to reason about, extend, and hand to agents without losing architecture.

## Purpose

This page is the default engineering checklist before changing a happydesigns repository.

## Decision rule

Prefer the smallest change that preserves the product boundary.

## Principles

- Reuse before rebuilding.
- Configure before forking.
- Centralize tokens and policies.
- Prefer Nuxt UI components for standard UI.
- Keep deterministic logic outside Vue when possible.
- Do not duplicate behavior between UI, API, MCP, webhooks, and jobs.
- Extract deliberately, not prematurely.
- Document boundaries before exposing extension points.

## Working style

If the same rule appears in multiple channels, extract the shared rule. If a change belongs to a product boundary, document the boundary before widening the implementation.

## Examples

- Add a semantic token before adding raw light/dark color classes to many components.
- Add an application service before duplicating a Business OS action across API and MCP.
- Add a docs rule when repeated agent mistakes show that architecture intent is unclear.
