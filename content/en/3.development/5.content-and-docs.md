Docus is used for durable human and agent documentation.

Docs should turn stable decisions into reusable knowledge. They should not become a place where every task note, implementation scratchpad, or customer tutorial is published.

Publish durable operating knowledge. Keep temporary work state outside published docs.

## Publishing standard

Published docs should be durable and reader-facing:

- Reader-facing.
- Stable enough to guide repeated work.
- Useful to both humans and agents.
- Focused on product, architecture, workflow, and durable decisions.

They should not become a public scratchpad:

- Temporary implementation logs.
- `.agents/` scratch work.
- Unreviewed planning notes.
- Raw task state.
- A full copy of external product docs.

Business OS can store knowledge-base registry metadata around Docus spaces. That does not mean operational scratch notes should become public documentation.

## Product docs

Product-specific deep docs belong on dedicated product documentation sites. This docs site should explain how those products fit together.
