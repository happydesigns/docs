Docus is used for durable human and agent documentation.

## Published docs should be

- Reader-facing.
- Stable enough to guide future work.
- Useful to both humans and agents.
- Focused on product, architecture, workflow, and durable decisions.

## Published docs should not be

- Temporary implementation logs.
- `.agents/` scratch work.
- Unreviewed planning notes.
- Raw task state.
- A full copy of external product docs.

Business OS may later store knowledge-base registry metadata around Docus spaces. That does not mean operational scratch notes should become public documentation.

## Product docs

Product-specific deep docs may later live on dedicated sites such as `ui.happydesigns.de`, `id.happydesigns.de`, or `os.happydesigns.de`. This docs site should explain how those products fit together.
