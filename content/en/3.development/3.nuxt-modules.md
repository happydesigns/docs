Nuxt modules are used for reusable Nuxt behavior that an app installs and configures.

Use a module when several Nuxt apps need the same setup, or when a package needs to participate in Nuxt itself through hooks, runtime config, generated files, generated types, or server handlers.

Do not put one customer's page composition, copy, brand treatment, or project-only dashboard work in a module. That work usually belongs in the customer project, a Nuxt layer, or content.

## Use modules for

- Registering hooks.
- Adding server routes.
- Defining runtime config.
- Generating files.
- Generating types.
- Adding server handlers.
- Providing behavior that is reusable across Nuxt apps.

## Examples

- `happydesigns/nuxt-variants` is a module because any supported app can install the same typed layout variant system.
- A reusable content-source integration can be a module if multiple apps install it with the same options.
- A customer-specific dashboard layout belongs in the customer project or client layer.
- If several customers need the same dashboard customization mechanism, build the general mechanism in the product or a module, then configure it per project.
- Business OS integrations use modules or server contracts when the integration behavior is reusable. They do not belong in brand layers.
