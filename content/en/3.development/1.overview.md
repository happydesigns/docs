happydesigns development favors reuse, clear boundaries, and central configuration over local one-off fixes.

This section turns the architecture model into working practice for Nuxt, Docus, Nuxt UI, content, and shared logic.

Reuse before rebuilding. Configure before forking. Centralize policy before scattering local exceptions.

## Scope

This section owns the build practices that repeat across happydesigns repositories:

- How to decide between Nuxt layers and Nuxt modules.
- How to keep deterministic logic outside Vue when possible.
- How to write Docus docs that remain useful to humans and agents.
- How to use Nuxt UI through semantic tokens and central app config.

It does not replace product references or upstream framework docs:

- Product-specific API references.
- Full Nuxt, Docus, or Nuxt UI docs.
- Unstable implementation logs.
- Customer-facing tutorials.

## Examples

- Use `UTable` for row/column data before rebuilding tables with cards.
- Use a Nuxt module for installable behavior, not a brand layer.
- Put Docus authoring rules here, not in every product README.
- Put customer website editing instructions in customer help docs.
